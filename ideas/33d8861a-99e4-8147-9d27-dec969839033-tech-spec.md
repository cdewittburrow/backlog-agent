## Overview
Normalize RSS feed content in First Read so every story has a consistent `lede` field — approximately 2–3 sentences, HTML-stripped, ≤300 characters, ending at a sentence boundary. The normalization happens at parse time, in the feed ingestion layer. No new UI work is needed — the existing lede display will simply receive reliable data. Axios-style feeds (full article body in feed content) and standard feeds (first paragraph only) are the two primary cases; others are handled by a configurable `schemaType` per source.

## Architecture
```
Feed fetch → RSS parse → normalizeContent(item, schemaType) → Story object → store/display
```
- The normalization step is inserted between the raw RSS parse and the Story data model
- Per-source `schemaType` is declared in a static source config map (not auto-detected at runtime)
- `normalizeContent` is a pure function: `(RawFeedItem, SchemaType) → string`
- Output: a single normalized `lede` string, guaranteed to be non-null

## Stack + dependencies
- Language: TypeScript (matching existing codebase)
- HTML stripping: `striptags` (lightweight, no DOM dependency) or equivalent — no new heavy dependencies
- Sentence boundary detection: naive approach (split on `. `, `! `, `? `) — sufficient for news lede normalization
- No new network calls or external APIs

## Key implementation decisions
- **Parse-time normalization, not render-time:** Normalization runs once when the feed is fetched. Stories are stored with the normalized lede already computed. No normalization logic in UI components.
- **Per-source static config, not auto-detection:** Auto-detecting schema type on first fetch is fragile (feeds change format, new fields appear). A static map per source is explicit and auditable. When a new source is added, the developer must declare its `schemaType`.
- **Truncation at sentence boundary, not character boundary:** A hard character cutoff mid-sentence is worse than a slightly longer lede that ends cleanly. Truncate at the last sentence boundary before 300 characters.
- **Lede identical to headline:** If the normalized lede exactly matches the headline (case-insensitive), suppress the lede in the UI — show headline only. This is a display-layer rule, not a normalization rule.
- **Default on unknown schemaType:** If a source has no `schemaType` set, default to `'standard'` and emit a log warning. Do not throw.

## Risks + unknowns
- **Axios feed format may change:** The normalization for Axios specifically depends on their feed content field being the full article body. If Axios changes their feed format, the `axios_full_body` logic will produce wrong output silently. Mitigation: add a sanity check — if extracted lede is >1500 chars, log a warning.
- **Unknown sources:** New RSS sources added without reviewing `schemaType` will silently use `standard`, which may produce wrong output for non-standard feeds.
- **Sentence boundary detection edge cases:** Abbreviations (U.S., Dr., etc.) contain periods. The naive split approach will mis-truncate on these. For news content this is acceptable in v1, but should be documented as a known limitation.
- **Non-Latin character sets:** Sentence boundary detection based on `. ` / `! ` / `? ` does not work for Chinese, Japanese, or Arabic. Non-issue if all current feed sources are English-language.
