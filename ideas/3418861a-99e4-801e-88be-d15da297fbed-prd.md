## Problem

Fullstack's current level scale and question difficulty distribution are miscalibrated. A user who scores well on their first session lands near the top of the level ladder, which breaks the game-feel: levels should feel earned over many sessions, not achievable in one sitting. The root issue is two-fold — questions are drawn randomly across all difficulties with no progression, and the scoring ceiling is too generous, mapping high performance directly to high-level placement.

## Goal

Recalibrate Fullstack's difficulty curve and scoring model so that sessions feel progressively harder as they go, and so that even a perfect performance places the user near the bottom of level 2 rather than near the top of the overall scale. The product should feel like something you get better at over time, not something you can max out immediately.

## Users

Chase — the primary and likely sole Fullstack user. This is a personal learning and progression tool, so calibration is tuned to one user's experience rather than a statistical population.

## Requirements

**Must have**
- Questions within a single session are drawn from difficulty bands, not a flat random pool — early questions come from an easier band, later questions from a harder band
- The number of difficulty bands and the band boundaries are configurable (not hardcoded magic numbers)
- The scoring ceiling is capped so that a perfect session places the user at approximately the bottom of level 2 on the level scale
- The level scale itself is re-parameterized to make higher levels require substantially more cumulative XP than the current model allows

**Should have**
- Difficulty band assignment is derived from question metadata already present in the data model (e.g., a `difficulty` field), not from a separate classification step
- The transition between bands within a session is smooth — there is no visible "level up" moment mid-session, just a gradual increase in question hardness
- A tuning document or constants file documents the chosen band boundaries and scoring cap values so they can be revisited without digging into logic

**Nice to have**
- A dry-run or simulation mode that shows projected level placement for a given score without committing it, useful for validating the calibration
- Historical session scores are retroactively re-mapped to the new scale so past progress isn't wiped

## Out of scope

- Changing the question content itself — this is purely a selection and scoring mechanic change
- Adaptive difficulty that adjusts mid-session based on performance (this would be a future feature)
- Leaderboards or comparative scoring — Fullstack is single-user
- Changing the visual level-up experience (separate from the numeric rescaling)

## Open questions

- Does the current data model have a `difficulty` field on questions, or would band assignment require a new classification pass over existing questions?
- How many levels exist on the current scale, and what is the current XP-per-level distribution? The rescaling needs those numbers as a baseline.
- Should the band progression be fixed per session length (e.g., first 5 questions = easy, next 5 = medium, last 5 = hard) or proportional (first third, middle third, last third)?
- What should happen to a user's current level placement after the rescaling — reset to 0, remap proportionally, or leave unchanged?

## Decision log

- The trigger for this feature was Chase landing near the top of the level scale on what should have been an early, imperfect session — confirmed the scoring model is too generous
- The solution direction (banded difficulty + rescaled ceiling) was set by Chase in the original note, not derived from user research — treat the two levers as requirements, not options to evaluate
- Perfect score → bottom of level 2 is the explicit calibration target; this should be treated as a hard constraint, not a soft guideline
