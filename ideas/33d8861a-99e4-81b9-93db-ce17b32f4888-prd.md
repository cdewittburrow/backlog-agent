## Problem
First Read is currently text-only. Images in news feeds are a well-established signal for story importance and make content easier to scan. However, RSS feed image support varies widely — some feeds include images in standard fields, others bury them in media extensions, and some do not include them at all. Before designing any visual treatment, we need to know what data is actually available.

## Goal
Determine whether current RSS/API sources expose image data, document the findings, and if sufficient coverage exists, define a v1 visual treatment for story cards.

## Users
Chase, scanning First Read. Visual treatment should help with quick triage — not decorate for its own sake.

## Requirements

**Must have**
- Audit of all current feed sources: do they expose images, and in which fields? (`media:content`, `enclosure`, `og:image` via meta scrape, etc.)
- Decision: proceed with images (>= 60% of sources have usable images) or deprioritize
- If proceeding: display image as a thumbnail on story cards

**Should have**
- Fallback for stories without images (clean text-only card, not a broken image placeholder)
- Lazy loading so images do not block feed render
- Images cached locally to avoid re-fetching on scroll

**Nice to have**
- Source logo/favicon as a fallback when no story image is available
- Aspect ratio normalization (all thumbnails same size regardless of source dimensions)

## Out of scope
- Scraping article pages to find images not in the feed
- Full-bleed image layouts (keep it scannable, not editorial)
- Video thumbnails

## Open questions
- What percentage of current feeds include image data in-feed?
- Is image data already being fetched and discarded, or not fetched at all?
- What is the current card layout — fixed height or dynamic?

## Decision log
- Spec gates on the audit — if fewer than ~60% of sources provide images natively, defer this feature rather than building a fallback scraper
