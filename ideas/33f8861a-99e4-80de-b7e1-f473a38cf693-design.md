## Overview

Guest intake adds a pre-voting phase to Just Pick Already. The host creates a session, enables intake, and shares a link. Guests open the link in any browser — no app install, no sign-in — and submit suggestions. The host watches suggestions arrive, curates the list, then closes intake and starts the normal swipe voting round. The intake phase is clearly separated from the selection phase; guests who submit don't automatically become voters unless they choose to swipe.

## User flows

**Host flow — create session with intake:**
1. Host signs in (existing magic link flow)
2. On room creation, host toggles "Let guests suggest options" (off by default for backwards compat)
3. Room is created in `intake` status instead of jumping straight to voting
4. Host is shown a shareable intake link + QR code
5. Host sees a live list of submitted suggestions as they arrive
6. Host can edit option names, remove options, or add their own before closing intake
7. When ready, host taps "Close intake & start voting" — room status flips to `open`, normal swipe flow begins

**Guest submission flow:**
1. Guest opens intake link in browser (no install required)
2. Page shows: room title / context, a simple text input, and a submit button
3. Guest optionally enters their name (shown alongside their suggestion in the host view)
4. Guest submits one suggestion (or more, if host allows multiple per guest)
5. Confirmation screen: "Got it! [Name] has been added."
6. Guest sees a read-only list of all suggestions (or a hidden count, depending on host's visibility setting)
7. Guest can choose to join the voting round via a link shown after submission

**Host flow — curate and start voting:**
1. Host reviews suggestions in a list view within the room management screen
2. Each suggestion shows submitter name (if provided) and submission time
3. Host can rename, delete, or reorder suggestions
4. Host taps "Close intake & start voting" — intake link expires, voting opens

**Error / edge flows:**
- Guest submits after intake closes → friendly "This session is no longer accepting suggestions" message
- Guest submits a duplicate (case-insensitive match) → inline validation error before submit
- Guest submits empty input → submit button disabled until non-empty

## Information architecture

**New room status: `intake`**
- Sits before `open` in the room lifecycle: `intake → open → closed`
- Rooms created without intake skip directly to `open` (no change to existing flow)

**New DB table: `room_intake_submissions`**
- `id`, `room_id`, `option_name`, `submitter_name` (nullable), `session_id` (localStorage UUID for dedup), `submitted_at`

**Existing tables updated:**
- `rooms`: add `intake_enabled: boolean`, `intake_expires_at: timestamp` (nullable), `intake_max_per_guest: integer` (default 1)
- `room_options`: add `submitted_by_guest: boolean` to distinguish host-added vs. guest-added options in the host view

**New routes:**
- `/intake/[room_id]` — public, no auth — guest submission form
- No new authenticated routes; intake management lives within the existing room management screen

## Interaction patterns

**Intake link sharing:**
- Host sees a "Share intake link" button on the room screen
- Tapping opens a share sheet (native share API on mobile, clipboard copy on desktop)
- QR code shown inline — useful for in-person gatherings

**Guest submission form:**
- Single-page, minimal layout: room title as header, optional "What kind of options are we picking from?" context blurb (pulled from room activity type), name field, suggestion field, submit button
- On mobile, the suggestion input is focused immediately on load
- After submission, the form resets to allow a second submission if the host allows multiple per guest; otherwise it disables the input and shows confirmation state

**Host live view:**
- Suggestions appear in real time via Supabase realtime subscription
- Each card shows: option name, submitter name (or "Anonymous"), timestamp
- Inline actions: rename (tap to edit in place), delete (swipe or icon)
- Badge on the room card in the host dashboard shows pending submission count

**Intake visibility for guests:**
- Default: guests see a running list of all submitted suggestions after they submit
- Host can toggle to "Hidden until voting starts" — guests see only their own submission and a count ("4 suggestions so far")

**Transition to voting:**
- When host closes intake, active guest intake pages show: "Intake is closed — voting has started!" with a link to join the swipe round
- The intake link returns a 410 Gone equivalent ("This session is no longer accepting suggestions")

## Edge cases + empty states

- **No suggestions submitted:** Host can still close intake and add options manually before starting voting — fallback to existing flow.
- **Host closes intake while guest is mid-submission:** Guest sees a race condition message ("Intake just closed — your suggestion wasn't saved") rather than a silent failure.
- **Intake link shared before room is ready:** Link shows a waiting state if room isn't in `intake` status yet (shouldn't happen normally, but guard against it).
- **Guest submits multiple times beyond the per-guest limit:** Enforced via session_id + count check server-side; client disables the form after limit is reached.
- **Duplicate option names:** Case-insensitive dedup within a room; guests see inline error, host sees a merge prompt if they add a duplicate manually.
- **Intake expiry:** If `intake_expires_at` is set and passes, the intake link automatically shows the closed state; the host can manually close before expiry at any time.
- **Room with 0 options after intake + host curation:** "Start voting" button is disabled with "Add at least one option to start" tooltip.

## Open questions

1. Should guests who submit be automatically redirected to the swipe voting view when intake closes, or only if they click through? Auto-redirect is more seamless but may surprise guests who submitted and walked away.
2. Is a QR code the right sharing mechanic for in-person use, or is clipboard + SMS more practical for Chase's typical use case?
3. Should the intake form support adding an image or place link per suggestion (leveraging the existing Google Places integration), or is plain text only for v1?
4. How should the host be notified of new submissions — push notification, in-app badge, or only via the live view when they have the room open?
