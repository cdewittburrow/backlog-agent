## Overview

The difficulty calibration redesign is a mechanics and scoring change, not a visual redesign. From the user's perspective, sessions simply feel harder as they progress and the XP/tier display reflects a recalibrated scale. The primary design work is defining how the invisible band progression behaves, how the rescaled tier display communicates the change to existing users, and what admin tooling (if any) surfaces the tuning constants.

## User flows

**Normal session (Daily Five or Freeplay):**
1. User starts a session — no change to entry point or any visible UI
2. Questions are served in band order: earlier questions drawn from easier difficulty band, later questions from harder band — no visible transition, no "band level up" moment
3. Session ends — score screen shows XP earned and current tier placement on the rescaled scale
4. XP bar fills noticeably slower than before — this is intentional; the pacing should feel earned

**Level Set (placement quiz) after recalibration:**
1. User completes the 15-question Level Set
2. A perfect score places the user at approximately the bottom of tier 2 (Newcomer → Apprentice threshold) — visibly lower than the current behavior
3. Tier placement screen communicates this with the existing axolotl moment; no new messaging needed unless a placement explanation is added

**First login after rescaling (existing users):**
1. Existing users with high tier placement see their tier either remapped proportionally or left unchanged (decision deferred to tech spec)
2. If remapping occurs, no in-app notification is required — the change is a correction, not a punishment; the XP bar simply reflects the new scale on next load

**Admin tuning flow (nice-to-have):**
1. Admin accesses `/admin` — a section shows current band configuration and scoring cap as read-only values
2. Admin can edit band boundaries and scoring ceiling via a simple form; save applies immediately to future sessions
3. No live-session interruption needed; mid-session question draw is not affected retroactively

## Information architecture

No navigation or structural changes are introduced. The following data/config structures are added or modified:

**New constants or config table:**
- `DIFFICULTY_BANDS` — array of band objects defining the difficulty range and question count per position in a session. Example structure: `[{ position_start: 0, position_end: 4, difficulty_max: 2 }, { position_start: 5, position_end: 9, difficulty_max: 4 }, ...]`. Stored in a constants file; optionally admin-configurable in a later iteration.
- `SCORING_CONFIG` — flat object with `session_xp_ceiling` (hard cap on XP earned per session) and the six tier XP thresholds (replacing the current hardcoded values).

**Question metadata prerequisite:**
- Every question needs a `difficulty` value (integer, e.g., 1–5) for band assignment. If questions are missing this field, a backfill step is required before the banded selector can run correctly.

## Interaction patterns

**During session — invisible to user:**
- No progress indicator for difficulty band
- No mid-session notification ("questions are getting harder")
- Questions simply become more challenging — the user may notice, but the UI makes no announcement

**Post-session XP display:**
- Existing score/XP screen is used unchanged
- XP values are smaller per session than before (by design); the UI should not add explanatory copy calling this out — the game feel communicates it

**Tier display and XP bar:**
- The XP bar fills against the new tier thresholds; existing visual components are reused
- If an existing user's remapped tier is lower than before, the bar visually resets to the new position on next load — no modal or explanation needed

**Admin tuning panel (if built):**
- A read-only display of current band config and scoring cap under `/admin`
- Editable form fields for band boundaries and ceiling value — plain number inputs, save button
- No undo; admin-only access means no destructive-action confirmation needed

## Edge cases + empty states

- **Questions without a `difficulty` field:** Band selector falls back to drawing from the full unfiltered pool for that band position — session still works, but without proper difficulty progression. This should trigger a console warning so the gap is visible in logs.
- **Difficulty band depleted (too few hard questions):** If a hard-difficulty band has fewer questions than the band requires, selector draws from the next closest difficulty rather than failing. Session is never empty.
- **Perfect session places user above tier 2:** This is the primary calibration failure mode. The scoring cap must be a hard ceiling enforced after XP is calculated — not a guideline. A dry-run simulation that prints projected tier placement for a given score is useful for validating this before deploy.
- **Session shorter than band count:** Daily Five is 5 questions; Freeplay is variable. Band boundaries must be proportional (e.g., first 40% easy, last 40% hard) rather than fixed question-count positions, so the progression works regardless of session length.
- **Retroactive XP remapping edge case:** A user who has earned exactly the minimum XP for tier 6 (Full Stack Builder) under the old scale will be remapped to a much lower tier. The remapping logic must be tested against the full tier range, not just the midpoint.

## Open questions

1. Does the current `questions` table have a populated `difficulty` field for all 213 questions? If not, how should difficulty be assigned — manually by Chase, or inferred from question format and category?
2. Should band boundaries be fixed per session length (e.g., first 2 questions = easy, last 2 = hard in a 5-question Daily Five) or proportional? Proportional handles variable session lengths more cleanly.
3. What is the target number of sessions to reach tier 3? This calibration target defines the scoring cap's upper bound and should be agreed before the tech spec sets the constant.
