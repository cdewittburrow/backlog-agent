## Overview

Authentication confirmation emails sent by the Fullstack app contain hardcoded `localhost` URLs in the verification link. This means the confirmation link only resolves on the developer's local machine and fails for any other environment (staging, production, any other device). The fix is to ensure the app's base URL is sourced from an environment variable at email-generation time rather than from a hardcoded or defaulting value.

## Architecture

Auth confirmation emails are generated during the sign-up / password-reset flow. The verification link is constructed by combining a base URL with a token path:

```
{BASE_URL}/auth/confirm?token={token}&type={type}
```

The bug appears at the point where `BASE_URL` is resolved. If the env var is missing or not loaded, the auth library (e.g. Supabase Auth, Auth.js, or custom middleware) falls back to `http://localhost:{port}`, which is only valid locally.

The fix touches two surfaces:
1. **Server-side env var** — ensure `BASE_URL` (or equivalent: `NEXTAUTH_URL`, `SITE_URL`, `APP_URL`) is set correctly in each deployment environment.
2. **Email template / link builder** — ensure the code reads from the env var and does not have a hardcoded fallback.

## Stack + dependencies

- **Fullstack app** (framework unknown — likely Next.js or similar Node-based stack based on context)
- Auth library: Supabase Auth, Auth.js/NextAuth, or custom JWT email flow — exact library determines where the base URL is configured
- Email delivery: transactional email provider (e.g. Resend, SendGrid, Postmark) — the base URL issue is upstream of the email send, in link construction
- Deployment: hosted environment with env vars managed via platform dashboard (e.g. Vercel, Railway, Render) or `.env` files

## Data model

No schema changes required. This is a configuration bug, not a data model bug.

If the auth library stores confirmation tokens in a database, those records are unaffected. Tokens generated with a localhost link are effectively invalid in production but do not need to be purged — they will expire naturally per the existing TTL.

## Key implementation decisions

**1. Identify the env var name used by the auth library**

| Library | Env var |
|---|---|
| Supabase Auth (self-hosted) | `SITE_URL` in Supabase config + `NEXT_PUBLIC_SUPABASE_URL` |
| Auth.js / NextAuth | `NEXTAUTH_URL` |
| Custom | `BASE_URL`, `APP_URL`, or similar |

Locate which library is in use and which variable controls the base URL for email links. This is the single most important step.

**2. Set the env var in all non-local environments**

- Production: set via hosting platform dashboard
- Staging/preview: set separately — do not assume it inherits from production
- Value must include protocol and no trailing slash: `https://app.fullstack.xyz`

**3. Audit the link-construction code**

Search for the URL being assembled in code. Common patterns to grep for:
- `localhost`
- `BASE_URL`
- `NEXTAUTH_URL`
- `SITE_URL`
- `confirmationUrl`, `verificationLink`, `emailRedirectTo`

If the code has a hardcoded fallback like `` `http://localhost:3000${path}` ``, replace it with a thrown error or logged warning so misconfiguration is caught at startup rather than silently producing broken links.

**4. Test in a non-local environment before closing**

Trigger a real confirmation email in staging and click the link from a different device to confirm resolution. Do not close this bug based on local testing alone.

## Risks + unknowns

- **Unknown auth library** — the exact fix location depends on which library handles email link generation. Must be confirmed before making changes.
- **Multiple environments with different base URLs** — staging and production need separate values; a single fix to production env may not catch staging.
- **Preview/branch deployments** — if the hosting platform spins up ephemeral preview URLs per branch, `NEXTAUTH_URL` or equivalent needs to be dynamically set (some platforms do this automatically, others require configuration).
- **Existing users with broken confirmation links** — users who signed up while the bug was active have unverifiable emails. They will need to re-trigger the confirmation flow after the fix is deployed. Consider whether to send a bulk re-confirmation or handle on a case-by-case basis.
- **HTTPS vs HTTP** — ensure the base URL in the env var uses `https://` in production, not `http://`. Some auth libraries will reject insecure redirect URLs.
