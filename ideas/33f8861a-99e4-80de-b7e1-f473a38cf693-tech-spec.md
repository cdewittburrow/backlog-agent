## Overview

Guest intake extends Just Pick Already with a pre-selection phase. A new `intake` room status sits before `open` in the room lifecycle. A public, no-auth web route (`/intake/[room_id]`) lets guests submit option suggestions via any browser. The host watches submissions arrive in real time via a Supabase realtime subscription, curates the list, then closes intake and starts the normal swipe voting round. All changes are backwards compatible — rooms created without intake enabled skip directly to `open` as before.

## Architecture

**New components:**
1. **`/intake/[room_id]` route** — public page served by the existing Vite/React app via React Router. No auth required. Reads room metadata, handles submission, and shows live or hidden suggestion list depending on host setting.
2. **`room_intake_submissions` table** — stores guest suggestions with submitter name, session identity, and timestamp.
3. **Realtime channel for host** — Supabase realtime subscription on `room_intake_submissions` filtered by `room_id`, used in the existing room management screen.
4. **`close_intake` RPC** — a PostgreSQL function that atomically copies submissions to `room_options` and flips `rooms.status` from `intake` to `open`.

**No new backend service** — Supabase handles storage, auth, RLS, and realtime. Vercel handles routing. No additional infrastructure needed.

## Stack + dependencies

- **Existing:** React + Vite, Tailwind CSS, Supabase JS client, React Router (verify it is already installed; if not, add `react-router-dom`)
- **New dependency:** `qrcode.react` (~10 KB) for inline QR code generation on the host's intake sharing screen — lightweight enough to justify adding
- **No other new packages required** — native Web Share API handles link sharing on mobile; clipboard API handles desktop fallback

## Data model

**`rooms` table — new columns:**
```sql
ALTER TABLE rooms
  ADD COLUMN intake_enabled       boolean     NOT NULL DEFAULT false,
  ADD COLUMN intake_expires_at    timestamptz NULL,
  ADD COLUMN intake_max_per_guest integer     NOT NULL DEFAULT 1;
```

**`rooms.status` enum — add `'intake'`:**
```sql
-- If status is an enum type:
ALTER TYPE room_status ADD VALUE 'intake' BEFORE 'open';
-- If status is a plain text column, no migration needed beyond application-layer checks.
```

**New table: `room_intake_submissions`:**
```sql
CREATE TABLE room_intake_submissions (
  id             uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  room_id        uuid        NOT NULL REFERENCES rooms(id) ON DELETE CASCADE,
  option_name    text        NOT NULL CHECK (char_length(trim(option_name)) > 0),
  submitter_name text,
  session_id     text        NOT NULL,
  submitted_at   timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX ON room_intake_submissions(room_id);
CREATE INDEX ON room_intake_submissions(room_id, session_id);
```

**`room_options` table — new column:**
```sql
ALTER TABLE room_options
  ADD COLUMN submitted_by_guest boolean NOT NULL DEFAULT false;
```

**Row-level security:**
- `room_intake_submissions` INSERT: allow when the referenced room's `intake_enabled = true` AND `status = 'intake'` AND (`intake_expires_at` IS NULL OR `intake_expires_at > now()`) — no `auth.uid()` check required (public submissions).
- `room_intake_submissions` SELECT: allow for the room organizer (`auth.uid() = rooms.organizer_id`) and for the submitting guest matching by `session_id` (guest reads their own submissions only).
- `room_intake_submissions` DELETE: allow only for the room organizer.

**`close_intake` RPC (PostgreSQL function):**
```sql
CREATE OR REPLACE FUNCTION close_intake(p_room_id uuid)
RETURNS void LANGUAGE plpgsql SECURITY DEFINER AS $$
BEGIN
  -- Verify caller is the room organizer
  IF NOT EXISTS (
    SELECT 1 FROM rooms WHERE id = p_room_id AND organizer_id = auth.uid()
  ) THEN
    RAISE EXCEPTION 'unauthorized';
  END IF;

  -- Copy submissions to room_options
  INSERT INTO room_options (room_id, name, submitted_by_guest)
  SELECT p_room_id, option_name, true
  FROM room_intake_submissions
  WHERE room_id = p_room_id;

  -- Flip room status to open and close intake
  UPDATE rooms
  SET status = 'open', intake_expires_at = now()
  WHERE id = p_room_id;
END;
$$;
```

## Key implementation decisions

1. **Guest session identity** — On first visit to `/intake/[room_id]`, the page reads or creates `localStorage.getItem('intake_session_id')` using `crypto.randomUUID()`. This ID is sent with every submission for per-guest dedup and limit enforcement. It can be spoofed via incognito, which is acceptable for Chase's small-group use case.

2. **Duplicate detection** — Client-side: before submit, compare the new option name (`.trim().toLowerCase()`) against all existing `room_options` names and all existing `room_intake_submissions` for the room. Show inline error if matched. Server-side: no UNIQUE constraint at the DB level — the host may want to allow near-duplicates or handle merging manually.

3. **Per-guest submission limit** — Enforced client-side by counting existing submissions for the current `session_id`. Server-side: the INSERT RLS policy or a trigger can enforce `intake_max_per_guest` by counting rows matching `(room_id, session_id)` before allowing the insert.

4. **Intake expiry** — The intake page fetches room metadata on load and checks `status` and `intake_expires_at`. If the room is not in `intake` status (or expiry has passed), the page renders a closed/expired state. A Supabase realtime subscription on the `rooms` row is set up for guests already on the page so they see the closed state without a reload when the host closes intake.

5. **Backwards compatibility** — `intake_enabled` defaults to `false`. All existing rooms with `status = 'open'` are unaffected. Room creation adds an optional "Let guests suggest options" toggle (off by default). No migration of existing room data is needed.

6. **QR code** — Generated entirely client-side from the intake URL string using `qrcode.react`. The QR code is displayed inline on the host's room management screen when intake is active.

7. **Atomicity of close_intake** — The copy-submissions-to-options + status-flip is a single `SECURITY DEFINER` RPC call. If the DB function errors mid-execution, Postgres rolls back the transaction automatically, preventing partial state (submissions copied but status not updated).

## Risks + unknowns

1. **React Router presence** — The app must use React Router (or equivalent) to add `/intake/[room_id]` as a client-side route. If the current SPA relies on hash routing or has no router, this needs to be added. Verify before starting.

2. **RLS cross-table conditions** — The INSERT RLS policy on `room_intake_submissions` needs to join to `rooms` to check `intake_enabled` and `status`. Supabase supports this via `EXISTS (SELECT 1 FROM rooms WHERE ...)` in the policy expression, but it must be tested — cross-table RLS without an auth context can have subtle behavior.

3. **Google Places integration** — The design doc mentions a potential "place link" per suggestion. This should be explicitly out of scope for v1 since the Places API key is currently broken. Plain text option names only until the key is restored.

4. **`room_status` enum vs. text** — If `rooms.status` is a PostgreSQL enum, adding `'intake'` as a new value requires a migration and there is no rollback path once values are added to an enum. Confirm the current column type before migrating.

5. **Realtime channel limits** — If the project is on Supabase's free tier, concurrent realtime channels are limited. The host's room management screen would add one channel per active room. Verify this is within the plan's limits.
