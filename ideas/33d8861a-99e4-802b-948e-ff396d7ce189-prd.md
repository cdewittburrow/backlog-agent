## Problem
The Garden app currently requires manual input to schedule or trigger irrigation. Real soil conditions vary based on recent weather, plant type, and sun exposure — factors that a fixed schedule cannot account for. The user wants remote moisture sensors in the ground to feed actual soil data back to the app, which would then automatically trigger the Rachio smart sprinkler system when moisture drops below a threshold.

## Goal
Integrate remote soil moisture sensors with the Garden app so that irrigation decisions are driven by real soil data rather than fixed schedules. Rachio should be triggered automatically when soil moisture falls below a configurable threshold, and the user should be able to monitor sensor readings and irrigation events in the app.

## Users
Chase, managing a home garden with Rachio irrigation already in place. The core need is set-and-forget automation that responds to actual conditions — not watering on a schedule when it rained yesterday.

## Requirements

**Must have**
- Support for at least one specific remote soil moisture sensor (hardware selection is a key open question — see below)
- Sensor readings ingested into the Garden app (via direct integration, cloud API, or local hub)
- Configurable moisture threshold per zone: when moisture drops below threshold, trigger a Rachio watering run
- View current moisture readings per sensor/zone in the Garden app
- Log of irrigation events triggered by sensor data (what ran, when, why)
- Manual override: ability to trigger or suppress a Rachio run regardless of sensor reading

**Should have**
- Multiple sensors supported (one per garden zone)
- Alert or notification when a sensor goes offline or battery is low
- Historical moisture readings chart (even basic) so the user can see trends
- Guard against over-watering: do not trigger if Rachio already ran within N hours or if rain is forecast (via Rachio's built-in weather skip, if available)

**Nice to have**
- Per-plant moisture thresholds based on plant type in the existing Garden app data model
- Dashboard view showing zone status (dry / adequate / saturated) at a glance

## Out of scope
- Building custom sensor hardware — this uses off-the-shelf sensors
- Multi-user or multi-property support
- Full Rachio app replacement — this augments, does not replace, the Rachio interface
- Weather-based scheduling (Rachio already handles this natively; do not duplicate)

## Open questions
- Which sensor hardware? Candidates: Govee soil sensors (Bluetooth, limited range), Ecowitt + soil probes (WiFi/gateway-based, cloud API available), or DIY with ESP32 + capacitive probe + local MQTT. The choice drives the entire integration architecture.
- Does Rachio expose an API for triggering manual zone runs? (Rachio v3 API does — confirm current API availability and auth model.)
- Where does the Garden app run — mobile only, or is there a backend service that could poll sensors on a schedule?
- What is the current Garden app data model — does it have a concept of zones?
- Is there an existing hub or smart home platform (Home Assistant, etc.) that could act as a middleware layer?

## Decision log
- This is a significant scope increase — from software to hardware + software. Hardware selection should happen before any integration code is written.
- Rachio API integration is the likely implementation path, contingent on confirming v3 API supports programmatic zone triggering.
