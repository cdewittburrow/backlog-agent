## Overview
Add a client-side dismiss gesture to First Read story cards. Users swipe left (mobile) or click an X icon (web) to remove a story from their feed. Dismissed story IDs are stored locally and persist across sessions. Stories are filtered at render time using the dismissed list. No backend changes required in v1.

## Architecture

```
Feed Component
  ├── fetch raw stories (existing)
  ├── load dismissed store (new)
  ├── filter: stories.filter(s => !dismissed.has(s.id))  (new)
  └── render filtered feed

Story Card Component
  ├── swipe handler (mobile) → dismissStory(id)  (new)
  ├── X icon tap handler (web) → dismissStory(id)  (new)
  └── undo toast (4s window)  (new)

Dismissed Store
  ├── load(): read from localStorage / AsyncStorage
  ├── add(id): write { id, dismissedAt } entry
  ├── remove(id): delete entry (undo)
  └── prune(): remove entries older than 30 days  (called on load)
```

The dismissed store is a simple key-value map (`storyId → dismissedAt`). It is loaded once at app start and kept in memory; writes are synchronous to local storage.

## Stack + dependencies

- **Web:** `localStorage` (built-in, no new dependency)
- **Mobile:** `@react-native-async-storage/async-storage` — verify this is already a dependency before adding
- **Swipe gesture (mobile):** check if the codebase already uses `react-native-gesture-handler` or `react-native-swipeable`; prefer reusing existing gesture infrastructure over adding a new library
- **Toast:** check if there is an existing in-app toast/notification primitive; if not, a lightweight one-off component is acceptable here (do not pull in a full toast library for a single use case)

## Data model

**Dismissed store schema (localStorage key: `first_read_dismissed`):**
```json
{
  "story-id-abc123": { "dismissedAt": "2026-04-14T10:00:00.000Z" },
  "story-id-def456": { "dismissedAt": "2026-04-13T08:30:00.000Z" }
}
```

- Story ID: use the feed item's GUID if available; fall back to URL. Investigate what stable identifier each feed provides — this is the most important implementation question before writing any code.
- Auto-expiry: prune entries with `dismissedAt` older than 30 days on every load. Prevents unbounded storage growth.
- No server-side sync in v1. Device-local only.

## Key implementation decisions

1. **Filter at render, not at fetch.** Raw feed data is unchanged. The dismissed list is applied as a filter in the component layer. This keeps the fetch layer clean and makes undo straightforward.

2. **Story ID strategy.** The feed spec (Stage 2 open question) must be resolved before coding: is there a stable GUID per story, or is the URL the only identifier? URL is acceptable for v1 but breaks if a source re-publishes at the same URL.

3. **Undo via toast.** Dismiss is immediately applied to the rendered list. A 4-second toast allows reversal. After the window closes, the dismissed entry is committed to storage. This avoids a two-phase write while still giving the user an escape hatch.

4. **30-day auto-expiry.** Prevents storage growing indefinitely for a long-lived client. Chosen over no-expiry because dismissed stories from last month are almost certainly irrelevant anyway.

5. **No "show dismissed" UI in v1.** The design doc marks this as optional — cut it for initial implementation to keep scope tight.

## Risks + unknowns

- **Story ID instability:** If feeds don't provide stable GUIDs, URL-based IDs may mis-match on re-publication or URL canonicalization changes. Investigate each active feed source before finalizing the ID strategy.
- **AsyncStorage availability:** On some React Native setups, AsyncStorage is deprecated from core. Confirm the mobile platform's storage solution before coding.
- **Swipe conflict with scroll:** On mobile, a horizontal swipe-to-dismiss must not interfere with vertical scroll on the feed. Threshold and direction detection need careful tuning.
- **First render flash:** If the dismissed list loads asynchronously after stories are fetched, dismissed stories may briefly appear before being filtered out. Load dismissed list before rendering or show a loading skeleton.

## Effort estimate

- Dismissed store abstraction (load, add, remove, prune): **~2h**
- Web X-icon tap + dismiss logic: **~2h**
- Mobile swipe gesture integration: **~3h** (varies by existing gesture infrastructure)
- Undo toast: **~2h** (varies by existing toast infrastructure)
- Feed filter integration + first-render sequencing: **~2h**
- QA + edge cases (empty state, storage full, swipe/scroll conflict): **~3h**

**Total estimate: ~14h**
