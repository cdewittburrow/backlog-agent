## Overview

Recipe import is failing in the Recipes app. The failure mode is not precisely described — "kinda failed" suggests a partial or silent failure rather than a hard crash. This spec covers investigation approach, likely root causes, and fix path.

## Architecture

The recipe import pipeline likely involves:
1. User-initiated trigger (URL paste, share sheet, or manual entry)
2. Fetch or scrape of external recipe source
3. Parse and normalize recipe content (title, ingredients, steps, metadata)
4. Persist to local or cloud data store
5. Display confirmation and navigate to saved recipe

Failure could occur at any step. Steps 2–3 are highest-risk given reliance on external sources.

## Stack + dependencies

- External recipe sources (structured data via Schema.org `Recipe` type, or HTML scraping)
- Network layer (fetch / HTTP client)
- HTML parser or recipe extraction library (e.g., `recipe-scrapers`, a custom parser, or a third-party API like Spoonacular)
- Data persistence layer (local DB, Supabase, or similar)

## Data model

A recipe import record likely includes:
- `source_url` — origin URL
- `title`, `ingredients[]`, `steps[]`, `servings`, `prep_time`, `cook_time`
- `imported_at` — timestamp
- `status` — `pending | success | failed` (if import status is tracked)

If there is no `status` field on imported records, add one to support retry and error surfacing.

## Key implementation decisions

**Investigation order:**
1. Check network layer — is the fetch timing out, returning a non-200, or blocked by CORS/bot protection on the source site?
2. Check parser — does the source use Schema.org JSON-LD, microdata, or raw HTML? If the source changed its markup, extraction breaks silently.
3. Check data persistence — is the parsed recipe failing validation before write, or is the write itself erroring?
4. Check error handling — is the failure swallowed silently or surfaced to the user?

**Fix options (by root cause):**
- Network/bot block → add request headers (User-Agent, Referer) or route through a server-side proxy
- Parser mismatch → update selectors or JSON-LD path; consider falling back to a structured extraction API
- Validation failure → log the offending fields, loosen constraints or handle nulls
- Silent failure → add error boundary and user-facing error state with a retry CTA

**Retry mechanism:** If not already present, add an explicit retry button on failure with the last-attempted URL pre-filled.

## Risks + unknowns

- If the failure is source-specific (one site changed markup), fix is narrow; if it affects all imports, the issue is systemic
- Third-party recipe extraction APIs have rate limits and cost implications if used at scale
- No description of how import is triggered — URL paste vs. share sheet may have different failure surfaces

## Effort estimate

- Investigation and root cause identification: 1–2 hours
- Fix (assuming single-layer failure): 2–4 hours
- Fix (if parser needs significant rework): 1–2 days
- Add retry UX + error state if missing: 2–3 hours

Total likely range: **half day to 2 days** depending on root cause.
