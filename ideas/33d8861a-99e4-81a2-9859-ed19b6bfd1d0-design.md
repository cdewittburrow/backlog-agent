## Overview
We are adding a dismiss gesture to First Read story cards. Users can swipe (mobile) or tap a dismiss control (web) to hide a story. Dismissed stories persist across sessions. This is purely an interaction and state management feature — no new surfaces, no new navigation.

## User flows

**Dismissing a story:**
1. User is scanning the feed
2. User swipes left on a story card (mobile) — or taps an X/dismiss icon (web)
3. Card animates out (swipe-away or fade)
4. Story is removed from the visible feed
5. Dismissed ID is written to persistent storage

**Recovering a dismissed story (optional v1):**
1. User taps "Show dismissed" toggle (if implemented)
2. Dismissed stories reappear in-line, visually muted
3. User can tap "Restore" to un-dismiss

**New session:**
1. User opens First Read
2. Feed loads — previously dismissed stories are filtered out before render

## Information architecture

**Dismissed stories store:**
- Keyed by story ID (URL or feed GUID)
- Structure: `{ id: string, dismissedAt: timestamp }`
- Stored in: localStorage (web) / AsyncStorage (mobile)
- No server-side sync in v1

**Feed render layer:**
- Filter: `stories.filter(s => !dismissed.has(s.id))`
- Applied at render time, not at fetch time (raw feed data unchanged)

## Interaction patterns

**Swipe (mobile):**
- Swipe left reveals a red dismiss zone or triggers immediate dismiss
- Follows platform conventions (similar to iOS Mail, most RSS readers)
- Threshold: ~40% of card width to confirm dismiss, snap-back if released early

**Tap target (web / fallback):**
- Small X icon in top-right corner of card, visible on hover
- Tap triggers same dismiss logic as swipe

**Undo:**
- Brief toast: "Story hidden. Undo?" with 4-second window
- Undo restores the story to its position in the feed

## Edge cases + empty states

- **All stories dismissed:** show "Nothing left to read" empty state with a refresh option
- **Story re-published at same URL:** treated as the same story — remains dismissed
- **Storage full / unavailable:** dismiss works for session only, warn in console (no user-facing error)
- **Feed refreshes while dismissed list is loading:** stories flash visible then disappear — mitigate by loading dismissed list before first render

## Open questions

- Does First Read have a stable story ID per item, or is identification URL-based?
- Is there already a local storage abstraction in the codebase, or will this be the first use?
- Should dismissed stories auto-expire after N days? (Recommendation: yes, 30 days, to prevent storage bloat)
